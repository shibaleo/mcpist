## 判断と学び

### MCP Tool Annotations への移行

独自フィールド（`defaultEnabled` / `dangerous`）を廃止し、MCP仕様の `annotations` に一本化した。

迷ったのは「独自フィールドの方が柔軟では？」という点。しかし MCP仕様が安定期に入った今、独自拡張を維持するコスト（Console と Go Server の両方でマッピングが必要）が上回ると判断した。`readOnlyHint` から「デフォルト有効かどうか」を導出する方式にしたことで、Console と Server で同じ基準を共有できるようになった。

### Batch セキュリティ: エラーメッセージの設計

一番考えたのは「クライアントにどこまで情報を返すか」。

最終的に、クライアントには `"batch rejected: one or more tools are not permitted"` という曖昧なメッセージだけを返し、具体的なツール名や拒否理由はサーバーログ（Loki）にのみ記録する設計にした。

理由: Layer 1 (Filter) で非表示にしたツールを LLM が呼ぼうとしている時点で異常。ツール名を返すと、有効/無効の情報を攻撃者に漏らすことになる。「誰に向けた情報か」を区別するのがセキュリティのエラーメッセージ設計の基本だと改めて感じた。

### gRPC 移行構想の評価

Google Cloud ブログの参照先（Python MCP SDK PR #1936）が reject されていたのが決定的だった。さらに調べると、MCP の双方向メッセージングモデル（LSP由来）と gRPC の一方向RPCモデルが根本的に不適合であるという Adrian Cole の指摘に辿り着いた。

学び: 技術提案を評価するとき、ブログ記事だけで判断せず「公式SDKの動向」と「仕様レベルの前提条件」を先に確認する。今回は PR の状態を調べたことで、仕様側の議論（SEP-1319, SEP-1352）の全体像が見えた。

### Observability の設計判断

X-Request-ID を Worker で発行し、Go Server → Loki → DB まで一貫して伝播させる設計にした。MCPクライアントからのリクエストIDは受け取らない（MCPクライアントはアノニマス前提、MCP仕様にトレーシングヘッダの規定もない）。

`LogSecurityEvent()` に `maybe_attacked: true` ラベルを付けたのは良い判断だった。Grafana ダッシュボードでフィルタするだけで攻撃の兆候を検知できる。
